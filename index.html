Getting started with tests ~ a TDD bullpen for JavaScript

Getting started with tests seems hard because there are many small decisions to make along the way. This post aims to make project tests easier to start and to share. We'll walk through the creation of a boilerplate project (I call it a bullpen) that you can re-use for driving your JavaScript tests. All steps were learned the hard way while I worked through my where.js project January and February, 2014.

why tests matter

The long-established truth is that every issue discovered late costs more to fix, the later the more costly. Test suites make issues easier to fix.

When someone objects that tests are extra or unnecessary, remember that bitcoin investors lost everything because MtGox programmers thought so, too.

my current stack of choice:

  • github ~ well-known online repository. You'll need either git or msysgit from github
  • node.js ~ server-side runtime for JavaScript
  • testem ~ node.js socket-based test runner for cross-browser development by toby ho
  • mocha ~ very flexible JavaScript test runner for node.js and browser tests
  • travis ~ remote service that runs your tests each time you commit to github
  • rawgithub ~ proxy service for viewing static pages from your github repo directly in the browser

Note: I write all code with Notepad++. On Windows. Proudly. Occasionally I grumble, but I work with it because it's more prevalent among lay users and supervisors who have no time for "it works on my machine."

installs

You'll need github and node.js first. I'll leave the setup details as an exercise for the reader.

With github and node.js squared away, open your command line shell or git-bash terminal and install testem and mocha, globally, one at a time:

npm install testem -g
npm install mocha -g

Note: testem has a known issue running mocha globally on Windows. We'll avoid that by calling mocha programmatically in the node-suite.

create a github repo

create a new project on github

  • name it 'tdd-bullpen' or something similar
  • choose .gitignore file
  • choose a license like MIT
  • choose README
  • choose public, not private (show your work)
  • click Create Repository

create the local project directory with git files:

  • cmd line

    mkdir bullpen
cd bullpen
git init

  • open /tdd-bullpen/.git/config file, and add these lines, replacing YOUR_NAME and PROJECT_NAME with real values:

    [remote "origin"]
  url = git@github.com:YOUR_NAME/PROJECT_NAME.git
  fetch = +refs/heads/*:refs/remotes/origin/*
[branch "master"]
  remote = origin
  merge = refs/heads/master

  • save this file.

Back at cmd line, pull down the license, README and .gitignore files from github

git pull origin master

Now we're in synch.

node.js configuration

cd bullpen
npm init

Use npm init to create a package.json file at the root of the project. This utility prompts you at the cmd line for a title, description, and some other things. Just get through enough steps to create the file.

create /test dir and some files

mkdir test
touch test/mocha-tests.js
touch test/node-suite.js
touch test/browser-suite.html

add a mocha test

open test/mocha-tests.js and add the following boilerplate:

// mocha-tests.js

var bullpen;

if (typeof require == 'function') {
  // enable to re-use in a browser without require.js
  bullpen = require('../index.js');
}

describe('smoke test', function() {
  assert(bullpen() === 'bullpen');
});

open test/node-suite.js and add the following to load the mocha tests:

// test/node-suite.js

// This runs with mocha programmatically rather than from the command line.
// how-to-with-comments taken from https://github.com/itaylor/qunit-mocha-ui

//Load mocha
var Mocha = require("mocha");
//Tell mocha to use the interface.
var mocha = new Mocha({ui:"bdd", reporter:"spec"});
//Add your test files
mocha.addFile("./test/mocha-tests.js");
//Run your tests
mocha.run(function(failures){
  process.exit(failures);
});

npm test

add an entry in package.json as 

"scripts" : {
  "test": "node test/node-suite.js"
},

including the trailing comma if it's not the last entry.

Now try the test commmand to run the mocha tests on node.js 

npm test

Should throw an error complaining that the index file wasn't found.

create the code to be tested

touch index.js

add an entry in package.json as 

"main" : "index.js",

including the trailing comma if it's not the last entry.

open the index.js file and add the following IIFE

;(function () {

  // node.js
  if (typeof module != "undefined") {
    module.exports = bullpen;
  }
  // browser
  if (typeof window != "undefined") {
    !window.bullpen && (window.bullpen = bullpen);
  }

  function bullpen() {

    return 'bullpen';
  }

}());

Run the test commmand again 

npm test

Should see mocha's spec reporter listing the single passing smoke test.

travis

set the project to notify travis

Visit the travis profile page to set up travis to be notified by github every time you commit something to your repo. You will be prompted to sign in with your github credentials if you haven't done this before.

On the travis profile, visit the Accounts page (mouseover your gravitar at the upper right - the Accounts link will appear).

Find the bullpen repo in the list that appears. If it's not there, click the Sync Now to refresh travis from github. When you find your repo listed, click the Off button to the On state.

Now travis will run the npm test command, which points to node test/node.suite.js and notify you of failures every time you push changes to your repo on github.

create a .travis.yml file (leading dot included)

This file goes in the root of the project (/bullpen/.travis.yml). Enter the following for travis to run the node.js tests:

language: node_js
node_js:
  - "0.11"
  - "0.8"

You don't need multiple node.js versions. I add the 0.8 entry in case of API changes I've missed.

add the service hook for travis on github

go to your travis profile page select Configure Services find travis in the list that appears - it should be checked

add travis badge to the README file

[![Build Status](https://travis-ci.org/YOUR_NAME/YOUR_PROJECT.png?branch=master)](https://travis-ci.org/YOUR_NAME/YOUR_PROJECT)

check in our code so far

cd tdd-bullpen
git status
git add .
git commit -am 'first test; travis config'
git push origin master

Travis should notify you that the build passed by your email.

You can also go to the project page on github and visit the travis profile page via the travis badge image. There you can see the test run for the last commit.

standalone browser suites

To run mocha in the browser you'll need a javascript and a css file separate from mocha's node bundle.

Create a /vendor directory

cd tdd-bullpen
mkdir vendor/mocha

Fetch these files and save them in the vendor directory:

  • https://github.com/visionmedia/mocha/blob/master/mocha.js
    • /vender/mocha/mocha.js
  • https://github.com/visionmedia/mocha/blob/master/mocha.css
    • /vender/mocha/mocha.css

Open the test/browser-suite.html file and add the following

<!DOCTYPE html>
<html>
  <head>
    <meta charset="utf-8">
    <title>tdd-bullpen mocha browser-suite</title>
    <link rel="stylesheet" href="../../vendor/mocha/mocha.css" />
  </head>
  <body>
    <div id="mocha"></div>
    <script src="../index.js"></script>
    <script src="../vendor/mocha/mocha.js"></script>
    <script>mocha.setup('bdd')</script>

    <!-- get tests and start -->
    <script src="./mocha-tests.js"></script>
    <script>
      mocha.checkLeaks();
      mocha.globals(['bullpen']); // this is our global bullpen function
      mocha.run();
    </script>        
  </body>
</html>

Open this file in your browser to verify filepaths are correct and that the single test passes.

add rawgithub link to the browser suite on your README page

Add this markdown on the README

[mocha suite](https://rawgithub.com/YOUR_NAME/YOUR_PROJECT/master/test/mocha/browser-suite.html)

check in these changes

cd tdd-bullpen
git status
git add .
git commit -am 'browser suite; mocha files; rawgithub'
git push origin master

view the browser suites on rawgithub

Give it a moment, then view the project page on github and visit the rawgithub page via the new link.

AFTER ALL THAT, NOW COMES THE TDD PART

re-use browser suites with testem

add the following inline script block (between mocha.setup and start comment) to browser-suite.html

...
<script>mocha.setup('bdd')</script>

<script>
  if (location.href.indexOf('://localhost:7357') != -1) {
    // testem goes between test runner setup and the tests
    document.write('<script src="/testem.js">' + '<\/' + 'script>');
  }
</script>

<!-- get tests and start -->
...

try it with testem

Tell testem to run, forwarding the browser to the URL of our test page

testem -t ./test/mocha/browser-suite.html

You should see testem open in the shell waiting for tests.

Now open any browser and go to http://localhost:7357/test/mocha/browser-suite.html

You should now see the browser suite with results, and see that testem has printed the results in the shell.

Hit 'q' to stop testem.

configure testem to launch the node.js suite via custom launcher

Create testem.json and add the following

{
  "launchers": {
    "mocha" : {
      "command": "node ./test/mocha/node-suite.js"
    }
  }
}

Verify this works by starting testem from the command line

testem -l mocha

You should see the single test passing in the shell. Hit 'q' to stop testem.

configure npm scripts to run testem in the browser

Now we'd like to run both node.js and browser-suite in testem at the same time. To set that up, add the testem entry in the scripts section of your project's package.json file:

"scripts" : {
  "test": "node ./test/mocha/node-suite.js",
  "testem": "testem -l mocha -t ./test/mocha/browser-suite.html" 
}

Now start both node and browser suites with testem from the command line 

npm run testem

You should see the passing test in both node and browser.

You now have a TDD bullpen you can use for a boilerplate setup for your

Don't Over-Promise

Don't Over-Promise (a pun & a parable)

In Blog Post Number One, I referred to the Zen of Python's rules for keeping things simple, readable, explainable.

That all seems simple, easy to repeat, good advice to avoid endless mindloops. But it's easily forgotten where there are many concerns or people to address where the number of communications becomes combinatoric.

Consider the ES6 es-discuss debates regarding Futures/Promises start here.
The threads go on for several days, the parties trying to hammer out the nuances of a serviceable API for Futures/Promises, based on Promises/A, Promises/A+ (then-ables), JQuery already-in-the-wild Deferred/Promises, etc.

And that's just the syntax and semantics of Promises to be supported natively in the next big iteration of JavaScript.

On the implementation side, James Coglan argues in a lengthy essay that the Node.js team missed the boat when they evicted Promise implementations from the Node.js core. Yet the lengthy comments and code examples demonstrate how hard it is to understand when not to use Promises (e.g., "promises try to solve the wrong problem", "the confusion comes from the attempt to use promises to control flow of statements, not dependencies on values").

Drew Crawford argues in Broken Promises, that Promises are not a general solution but best applied to specific situations, because, for example, "a great number of perfectly ordinary things depend on knowing how many elements there are." Trouble arises when the number of elements now or in future is undetermined. Drew mentions several problems that must be addressed by Futures/Promises libraries - memory, composability, parallelization and atomicity. After walking through these he has created a new DSL "to specify control flow and timing information" – all things which the language already does for you… (then came enlightenment)

The Node.js team's decision, then, is not a vote against Promises, but a vote to keep things simple in the core and let harder issues be sorted out in userland.

If it is hard just to understand the problem, the solution will more than likely be hard to explain. If the implementation is hard to explain, it's a bad idea.

Start Simple

Start Simple

I like the Zen of Python, particularly the points excerpted by Jack Diederich in his talk, "Stop Writing Classes," at PyCon 2012:

  • simple is better than complex
  • flat is better than nested
  • readability counts
  • if the implementation is hard to explain it's a bad idea
  • if the implementation is easy to explain it may be a good idea

He also emphasizes:

  • prefer easy things
  • don't do hard things in the first place
  • revert complications later

And finally, all this in service of the following:

We ship features, we do not ship code. We don't have customers because we have lots of code, we have customers because we have lots of features.

Hello World

Blogging at last. Thanks for stopping by.

Updated 2013-08-14: I've switched from Octopress to Hexo, a Node.js alternative. Check the documentation to learn how to use.

Who Am I

I'm a JavaScript developer mainly on the front-end of web applications.

  • github repos
  • gists

I Like Test-Driven Development

I really like test-driven development. TDD helps me keep things simpler, flatter, readable, and explainable. If I don't understand where the tests are going or how to get there, then it's time to stop and get help or re-think the problem – mainly to avoid checking something in that will harm others' understanding and productivity.

The list of TDD advantages below is copied from a comment on Cedric Beust's blog:

  • Interruptability
  • Less fragility
  • Refactoring
  • Less time wasted

Things I Don't Like

As a front-end developer, there are just some things about development I don't like:

  • tagsoup
  • the culture of hacking and rushing (to borrow a tweet from Alan Cooper))
  • JavaScript's lack of global management (namespacing by object trees doesn't count)
  • JavaScript's lack of import management (CommonJS doesn't count in the browser; AMD doesn't count on Node.JS)
  • CSS's lack of variables
  • CSS's lack of modules
  • reliance on pre-processors to solve these issues
  • the in-fill rushing of experts solving these lacunae
  • expert opinions on performance
  • opinions in general

On the Fence

I'm Not Sure About Pre-processors:

  • JS preprocessors (CoffeeScript)
  • HTML preprocessors (jade, haml, markdown)
  • CSS preprocessors (sass, less, stylus, rework)
  • ANY preprocessors and source transformation steps (browserify, uglify, closure, compressor)

Disclaimer, of course

Being opinionated myself, and occasionally correct, I reserve the right to change my mind on these topics.
